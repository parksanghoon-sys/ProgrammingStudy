# GitFlow 란?

여러 장기 브런치- main, feature, develop, release, hotfix 등의 브런치를 사용하여 병렬 개발 및 버전 관리 소프트웨어를 관리하는 Git 분기 모델이다.

## 전략

* Main branch
  * 서버에 배포된 운영 준비코드
* Develop branch
  * 진행 중인 작업으로 릴리스 전 통합에 사용
* Feature branch
  * 각 기능 별로 한가지 브런치 (예 feature/user-auth)
* Release branch
  * 배포 준비, 최종 테스트 및 버그 수정
* Hotfix branch
  * 배포 현장에서의 긴급 수정

### 실제 Git flow 시나리오

예를 들어 고객을 위한 애플리케이션을 만들있을때, 사용자들은 버전 2.3,2.4을 사용중이며 최신은 3.0 버젼이 있을 경우 기업은 요구상항떄문에 여러개의 버젼을 동시에 운영해야하는 경우가 있다.

이제 `버전 2.3의 버그를 수정해야하지만, 이미 3.0 작업 중 일때`

Gitflow에서는 이과정이 단순하다

1. 테그 버전 2.3 이 브런치에서 hotfix 브런치를 생성한다
2. 버그를 고친다
3. 두가지를 모두 `main` 과 `develop`에 병합한다
4. 릴리즈 버전 2.3.1 로 다시 배포 및 테그한다.

### GitFlow가 의미가 있을 때

1. 데스크톱 애플리케이션, 모바일 앱, 고객이 자체 인프라에 설치하는 엔터프라이즈 소프트웨어 등 **다양한 버전으로 출시되는 소프트웨어**입니다. 예를 들어 Electron 앱, iOS/Android 앱, 설치 프로그램이 포함된 소프트웨어를 생각해 보세요.
2. **여러 버전을 동시에 지원하는 것** — 버전 3.x를 개발하면서 버전 2.x를 유지하고 있다면, GitFlow는 이를 처리할 수 있는 구조를 제공합니다. 3+ 동시 버전을 지원하는 팀이 가장 큰 혜택을 봅니다.
3. **명확한 구조가 필요한 대규모 팀** — 릴리스와 핫픽스를 명확히 분기하면 개발자가 20+ 명일 때 혼란을 방지할 수 있습니다. 제 경험상 50+ 개발자 팀은 거의 항상 이런 수준의 구조가 필요합니다.
4. **규제받는 산업** — 의료, 금융, 정부 — 모든 변경 사항이 승인, 감사 추적, 추적성이 필요한 곳
