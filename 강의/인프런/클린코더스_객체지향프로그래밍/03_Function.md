# Function

### 원칙

* 함수의 책임은 하나만 지켜야한다.
* while , if ... 등은 없는것이 좋다.
* 잘 지어진 서술적인 긴 이름을 갖는 작은 함수로 유지해야한다.

### Rule

* 더 이상 작아질수 없을만큼 작아야한다
* 큰 함수를 보면 클래스로 추출할 생각을 해야함
* 클래스는 일련의 변수에 동작하는 기능의 집합

### 예제 깃 주소

[fitness example](https://github.com/msbaek/fitness-example)

### 개선

* 읽기 쉬어 지고
* 이해하기 쉬어지고
* 함수가 자기의 의도를 잘 전달

**개선의 원인**

* 함수는 작아지는 한 작아져야함
* 블록이 적어야함
* 들여쓰기는 한 두단계 정도만

## 목표

**큰 함수는 들여쓰기에 존재하고, 변수들을 사용해서 통신하는 기능의 집합이므로 클래스로 분리가 가능하다**

함수는 작은단위로 나뉘어야한다.

함수의 이름은 해당 함수의 목표를 표출을 잘해야한다.

## 테스트 작성법 (예제)

* 테스트 파일이 없는 계산 프로그램의 경우
  * 일단 기본 값 을주고 실행한후 출력을 확인한다.
  * 파일을 만들어 그파일과 실행후 파일을 라인비교를 통해 확인한다.

**One thing**

* 함수는 한가지만 해야한다.
* Caller 입장에선 한번만. Reader 입장에서는 One thing이 아니다.
* 큰 함수를 작은 함수로 쪼갤때 흥미로운 일을 수행
  * 함수를 서로 다른 추상화 레벨로 분리
    * 함수가 하나 이상의 추상화 레벨으 다루면 이 함수는 한가지 이상의 일을 한다.
* 이름을 잘지어라
  * 당신의 코드를 이해하기위한 네비게이션 역할을한다.
* 체크 리스트
  * 함수가 한가지 일만 하는지 확실한가?
  * 더이상 추출할 수 없을때 까지 추출하라
  * 추출 할 코드를 가진 함수는 한가지 이상의 일을 하는 것이다
  * 4줄 이내의 함수로만 구성된 클래스
  * if , while등 에서 {} 가 보인다면 extract 대상이다
