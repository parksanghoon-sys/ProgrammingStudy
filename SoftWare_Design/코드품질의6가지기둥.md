## 코드 가독성 향상

코드는 기계가 실행하기 위해 작성이 되는것이 아니다. **인간이 이해할 수 있도록 작성되어야 한다는 것이다.** 다음은 가독성을 높이기 위한 주요 요점이다.

* "원 스크린" 규칙
  * 일반적인 함수나 메서드가 화면높이( 약 20줄 코드)를 **넘지 않아야 한다**
  * 메더드가 여러페이지에 걸쳐있으면 논리를 추적하고 무슨일이 일어나느지 추측하기가 힘들어진다.
* 의미 있는 명명
  * 제공하는 **비지니스 로직**을 반영해야한다. 이것은 변수와 메서드로 부터 모든 명명 규칙에 적용된다.
    * 코드
      * 모호한 이름대신, 정확히 무엇을 하고 왜 그런지 알려주어야한다.
    * 인프라
      * 매핑 문서가 없으면 개발자용인지, 프로덕션용인지, 어느 서비스에 속하는지 아무도 모른다. 한눈에 맥락을 제공하는 이름을 사용한다.
* 소규모 방법과 단일 책임
  * 방법을 간결하게 유지하고 **단 하나의 책임**에 집중
    * 함수가 너무 많은 작업을 수행한다면, 논리를 더 작은 보조 메서드로 추출

> 이상적으로 코드가 **자체 문서화** 가 되어야한다. 의도를 명확히 전달하는 표현적 메서드 이름을 선택함으로서, 논리는 발전하지만 과거 주석이 머무르는 '주석- 코드 불일치' 위험을 제거 할 수 있다.

## 결정론을 목표로 하자

고품질의 코드를 작성하려면 함수들이 예측 가능해야 한다. 같은 입력을 제공하면 항상 같은 출력을 얻을 수 있다. 이 예측 가능성이 없으면, 코드를 변경하지않고 때로는 통과하고 어떤 테스트는 실패하는 '불안정한 테스트' 를 격게된다.

> Side Effects(부작용) 은 함수가 단순히 값을 반환하는 것 이상을 할때 발생한다. 국지적 범위를 벗어난 무언가를 수정하는 것이다.

**Side-Effects의 원인**

* 상태 돌연변이
  * DTO 내의 속성을 변경하거나 전역변수를 수정하는것, 참조 통과와 값 통과의 차이를 이해하는것이 의도지 않은 데이터 변조를 방조하는데 중요하다
* 비결정론적 기능
  * GUID 처럼 지속적으로 변하는 값을 사용한다, 이로인해 함수가 제대로 추상화되거나 주입되지 않은 한 일관되게 테스트할 수 없다. `DateTime.Now, Math.Random()`
* 부적절한 널 헨들링
  * 함수가 수신하거나 반환할 때 제대로 처리하지 않는다면 '널 참조 예외'가 발생한다. 호출자가 크래시 되지 않도록 반환할 명확한 계약을 정의해야한다.
* 숨겨진 의존성
  * 환경변수에 의존하거나 예상치 못한 예외를 던지는 함수는 추적하기 어려운 숨겨진 부작용을 만든다.
* 외부 입출력 시스템
  * 외부 세계와 모든 상호작용 ex) 파일읽기/쓰기, API 호출, 데이터베이스 쿼리 등.. 이것은 본질적으로 '부작용'이며 , 핵심 비지니스 논리와는 분리되어야한다.

## 코드를 요용하기 어렵게 만들기

훌륭한 엔드포인트는 단순한 메서드든 잘못된사용에 강해야한다. 

* 질못된 상태를 표현할 수 없게 만들어라
  * 타입 시스템을 사용하여 비지니스 규칙을 강제해라
* 빠르고 크게 실패하자
  * 컴파일 타임 : 개발 중 오류를 포착하기
  * 런 타임 : 실행중에 유효성 검사에 실패하면, 즉시 명확하고 구체적인 오류 메시지를 표시해라
* 안전한 기본값제공
  * 사용자가 특정 구성을 제공하지 않는경우, 시스템은 가장 위험한 옵션이 아니라 가장 안전한 옵션으로 기본값을 설정하라
* 외부에 드러나는 인터페이스를 작게 유지하자, 사용하는 사람이 이해해야 할 양을 줄여라
  * 캡슐화 : 내부 로직을 숨긴다
  * 패턴 : 복잡한 객체를 생성 시에는 팩토리나 필더를 사용하고 사용자가 복잡한 과정을 쉽게 처리하도록 하자
* 명시적인 처리가 바람질하다.
  * 무슨일이 일어나고 있는지를 명확하게 해야한다, 배경에서 숨겨져서 일어나는 "특별한 처리" 나 비밀스러운 동작은 피해야한다.
* 변화를 숨기자
  * 객체의 상태변화를 외부 세계로부터 숨기자, 불변객체나 트랜잭션 기반의 업데이트 방식을 선호해야한다.
  * 객체가 생성된 이후에는 변경될수 없다면, 공유된 상태와 관련된 오류를 방지할 ㅅ ㅜ있다.
* 타입을 문서보다 선호하자
  * 문서는 시간이 지나며 변경될 수있다. 주석또한 마찬가지다
  * 명확한 메서드나 변수의 타입이 잘 명시되어있으면 별도의 설명서가 필요 없다.

## 코드를 모듈화 하자

깔끔하게 작성된 함수를 만드는것은 시작에 불과하다, 중요한 것은, 그 함수들을 어떻게 체계적으로 조직화 하는것이다.

복잡성을 줄이기 위해 모듈화해야한다.

* 적절한 아키텍쳐를 선택하자
  * 모듈화 방식은 시스템의 규모에 따라 달라진다.
    * 계층적 아키텍쳐
      * 전통적인 방식이다 (Controller -> Service -> Repository -> DTO) 소규모이고 단순한 어플리케이션에 적합
    * 클린아키텍쳐와 모듈러 모놀리스
      * 성장하는 시스템을 위한 최적의 솔루션이다.
      * 이방식을 통해 핵심 비지니스 로직을 외부프레임워크나 UI와 분리한다.
    * 분산 시스템
      * 대규모 환경에서는. 복잡한 데이터 흐름을 처리하기 위해 머이크로서비스 아키텍쳐, 이벤트 기반 아키텍쳐, 또는 CQRS 와 이벤트 소싱기술을 고려
* 자신의 한계를 정하자
  * 코드를 부닐하는 것은 단순한 파일을 옮기는 것이 아니라, 논리적인 구분을 만드는 것이다.
  * Domain- Driven Design(DDD)
    * DDD 를 확용하여 핵심 도메인과 관련된 범위를 명확히해라
    * API 계약과 데이터 소유권이 어디서 겹치는지 살펴봐라
    * 만약 여러서비스가 동일한 데이터를 두고 경쟁한다면, 시스템의 경계가 잘못 설정된 것일 수 있다.
  * Separation of Concerns(SoC)
    * 관심사의 분리(SoC) 코드를 논리적인 기능에 따라 분리해야한다. 즉 UI, 어플리케이션 로직, 도메인관련 기능, 인프라 관련 기능 등으로 나누는 것이다.
    * 높은 응집도를 유지하고, 부분간의 결합도는 최소화 해야한다.
  * Single Responsibility Principle (SRP)
    * 단일책임우너칙 이 원칙은 메서드 뿐아니라 모듈과 클래스에도 마찬가지로 적용되어야 한다
    * 각 모듈은 하나의 명확한 책임을 가지며, 변경될 이유도 단 하나뿐이여야한다.
  * Dependency Injection (DI)
    * 의존성 주입(DI)를 사용하면 코드를 "플러그인 가능한" 형태로 만들 수 있다/
    * 이를 통해 복잡한 설명 없이, 단위 테스트도 훨신 수월해 진다. 왜냐하면 인프라를 모의 객체로 쉽게 대체가 가능하기 때문이다.
  * Keep It Simple Stupid && You Ain't Gonna Need It (KISS & YANGNI)
    * 불필요하게 복잡하게 만들지 마세요
    * 간단한 패키지 기반의 시스템으로도 충분한 경우에는 분산형 시스템을 만들필요가 없다
    * 비지니스 요구사항이 실제로 그러한 복잡성을 요구할 때까지는 복잡성을 피하는 것이 좋다.

> 모듈화란 단순히 시스템을 구성하는 방식에 관한것이 아니라, 인지적 부담을 줄이는데 관련이 있다. 개발자는 각각의 문제를 분리합으로써, 전체 시스템이 붕괴될까봐 걱정하지 않고, 한번에 하나의 요소에만 집중이 가능하다.

## 코드를 재사용할 수 있도록 만들자

재사용성을 높이는 목적은 단순히 "복사-붙여넣기"를 줄이는 것이 아니다, 오히려, 시스템 전반에 걸쳐 유사한 문제를 해결할 수 있도록, 견고하고 테스트가 잘되어 있으며, 범용성이 뛰어난 구성요소를 만드는것이 목적이다.

* 작고 특정한 구성요소들
  * 함수, 메서드, API 등 가능한 작고 목적이 명확하게 유지되어야한다. 한가지 기능만을 수행하는 함수는 "유틸리티"나 공통적으로 사용되는 컴포넌트가 되어, 시스템의 다른 부분들이 필요에따라 쉽게 활용이 되어야한다.
  * 이렇게 하면 불필요한 복잡성을 줄일 수 있다.
* 제네릭과 매개변수화
  * 만약 서로다른 타입에 대한 동일한 로직을 반복해서 작성해야한다면, 이제는 제네릭을 사용할때이다. 제네릭을 사용하면 한번만 로직을 작성하고, 나중에 데이터 타입을 파라미터로 지정할 수 있다.

## 코드를 테스트할 수 있도록 만들고, 테스트를, 제대로 해야한다.

'제대로 작동하는' 코드를 작성하는것은 일의 절반이다, 나머지 절반은 그 코드가 테스트 가능한지를 확인하는 것이며, 무엇보다도 실제로 테스트가 이루어졌는지를 확인하는 것이다.

코드 커버리지는 높지만, 테스트 내용이 의미있는 내용이 전혀 없는 프로젝트를 접할수 있다. 주장이나 검증이 없는 높은 코드 커버리즈는 '잘못된 안도감'에 불가하며, 결국 소용이 없다. 신뢰할 수있는 테스트세트를 구축하려면 다음의 원칙을 명심하다.

* 모든 방법은 가장 작은 단위로 나누어 분석하기
  * "작은 함수들은 테스트하기 쉽다" 만약 어떤 메서드가 너무 많은 일을 한다면, 단 하나의 시나리오를 테스트하기 위해서도 노력이 필요하다. 메서드를 간단하고 명확하게 유지함으로써, 특정 로직들을 개별적으로 테스트 가능하다.
* 의존성 주입을 통해 결합도를 낮추자
  * 의존성 주입을 활용하여, 핵심로직과 외부의존성들을 분리해라
  * 이렇게 하면 테스트 시 모의 객체를 사용할 수 있으며, 단위 테스트가 인프라가 아닌 코드 자체를 테스트하고 있음을 보장한다.
* "정상적인 경우" 와 "예외적인 경우" 모드를 테스트해야한다.
  * 코드가 모든 상황에서 동작하는지만을 테스트하는것은 충분하지 않다 견고한 테스트라면 다음과같은 경우도 반드시 포함하라
    * 비지니스관련 오류사례
      * 시스템은 자금 부족이나 제고 부족같은 상황을 어떻게 처리하나?
    * 경계값
      * 한계점에서는 어떤일이 일어나는가, 0, 음수 또는 허용 최대값을 사용하여 테스트하라
* 테이블 기반 테스트를 사용하라, 즉, 입력값과 예상되는 출력랎이 포함된 테이블을 정의한뒤, 하나의 테스트 함수 내에서 그 테이블의 모든 데이터를 순회하면서 테스트를 수행한다
  * C# - DataRow (MsTest), TestCase(Nunits), InineData(Xunit)

## 결론

* 코드를 모듈화하고, 읽기 쉽게 만들라, 때로는 코드를 너무 세분화하면, 수많은 파일이 생기게되는데 그런경우 코드를 읽는 사람은 여기저기로 이동해야 하며, 이는 인지적 부담을 유발한다, 이런경우 '3번이상 반복되는 경우에만 코드를 분리하라'라른 원칙을 적용이 가능하다
* 성능/ 가독성/ 테스트 가능성이라는 관점에서 보면, 특정 작업을 위해 최적화된 코드는 오히려 읽기 어렵거나 테스트가 어려울 수있다.

이러한 요소들을 잘 조화해야한다.
